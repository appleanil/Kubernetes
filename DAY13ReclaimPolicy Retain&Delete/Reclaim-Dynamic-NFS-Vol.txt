Reclaim policies --> Default Reclaim policy
--------------------------------------------
1. Retain: When a PV is released from a PVC with a "Retain" reclaim policy, the volume is not automatically deleted. Instead,it remains in the cluster in a released state.
            The underlying storage is preserved, allowing you to manually reclaim or manage the data.
My mongo pod and PVC is deleted, PV is in released state and data is safe(NFS-Server).
For long time we have data then use "Retain"

  
2. Delete: With the "Delete" reclaim policy, when a PVC is deleted, the associated PV is also deleted, along with the underlying storage.This means that all data is lost.
   My mongo pod and PVC is deleted, PV is in Failed state and data is not safe(NFS-Server).
   For short term or sanitary check use "Delete"

 3. Recycle(Deprecated): The "Recycle" policy allowed for a PV to be scrubbed (data deleted) and then made available again when released.
                          However, this feature has been deprecated in favor of more explicit data management practices. 

Example: PersistentVolume
-------------------------
apiVersion: v1
kind: PersistentVolume  
metadata:
  name: mongodb-pv   # here we do use namespace bcz its cluster level object kubectl get pv
spec:
  capacity:
    storage: 1Gi     # Adjust as necessary
  accessModes:
    - ReadWriteMany  # NFS supports this
  nfs:
    server: 172.31.40.109  # Your NFS server IP
    path: /mnt/nfs_share   # Path on your NFS server
  persistentVolumeReclaimPolicy: Delete  # while practicing we need to delete this line, later use this line or depending on your use case

PersistentVolumeClaim
---------------------
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mongodb-pvc  # should be same as mongodb spec, volumes persistentVolumeClaim: mongodb-pvc 
  namespace: prod    # This is correct
spec:
  accessModes:
    - ReadWriteMany  # Must match the PV's access mode
  resources:
    requests:
      storage: 1Gi   # Must match or be less than the PV's size

Dynamic volumes with nfs  
========================
Dynamic provisioning with NFS allows Kubernetes to automatically create PersistentVolumes using a StorageClass and NFS provisioner when a PVC is requested,
no need to create manually PV. And by default in dynamic vol with nfs Reclaim policy is "Delete"
In this also we loss the data just for short time use, but real time use only Retain.

Assignment --> bring our springwebapp and insert some data and later delete the PV(nfs-server) plz check out data is safe?
Answer ------> data is safe

+-------------------+-----------+-------------------+
| Component         | Deleted?  | Data Impact       |
+-------------------+-----------+-------------------+
| Provisioner       | Yes       | No impact         |
| StorageClass      | Yes       | No impact         |
| PVC               | No        | Data still safe   |
| NFS Server Data   | No        | Safe              |
+-------------------+-----------+-------------------+        


NFS Provisioner
Runs as a pod
Talks to NFS server
Creates volumes dynamically
No PV defined manually

When PVC is deleted:
NFS folder is deleted
PV is deleted automatically

Earlier: --------------> Manual PV → Failed.

Now: Dynamic PV ------->  auto managed and No Failed state issues.

Advantages
----------
| Feature              | Benefit            |
| -------------------- | ------------------ |
| No manual PV         | Saves time         |
| Auto scaling storage | Easy               |
| Works with RWX       | Multiple pods      |
| Less errors          | No mismatch issues |
